HANDOFF_PACKET — Post-Onboarding Email Gate Modal Freeze Fix
Date: 2026-03-20

SUMMARY

Fixed app freeze on first post-onboarding landing. The email verification gate modal was presenting during an in-flight navigation transition, leaving its touch responder broken on iOS. Buttons were unresponsive, requiring force-close. Fix defers modal presentation using InteractionManager.runAfterInteractions so the modal only appears after all navigation transitions and the calendar initial mount are fully settled.

PROBABLE ROOT CAUSE (static analysis only)

After onboarding, routeAfterAuthSuccess calls rebootstrapAfterLogin which sets globalStatus to authed BEFORE calling router.replace /calendar. The email gate effect in BootRouter fires immediately on the authed transition and starts an 800ms setTimeout. This timer fires while the calendar screen heavy initial mount is still in progress. Presenting a React Native Modal during an in-flight navigation transition on iOS can leave the modal touch responder improperly configured, making the I will do it later and Resend email buttons unresponsive.

MODAL ANALYSIS

Modal component: src/components/EmailVerificationGateModal.tsx line 31
Dismiss handler: src/components/EmailVerificationGateModal.tsx line 56 handleDismiss calls onClose prop
Condition that triggers display: src/app/_layout.tsx lines 654-720 email gate useEffect fires when bootStatus is authed AND session.user.emailVerified is false AND hasShownGateModal returns false for the userId
What is different about first post-onboarding render vs normal launch: On normal launch hasShownGateModal returns true because it was marked on first presentation. On first post-onboarding render the modal has never been shown, AND the timer starts during the rebootstrap-to-calendar navigation transition window.
Definitive bug vs defensive fix: Defensive fix. The dismiss handler code is correct. The probable cause is a timing issue where Modal is presented during an in-flight navigation causing iOS touch responder misconfiguration. InteractionManager.runAfterInteractions ensures the modal only presents when the UI is idle.
