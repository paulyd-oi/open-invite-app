SUMMARY

Fixed Profile "Open Invites" section showing "Invalid Date" event cards and dead-end navigation when tapping them. Root cause: JS Invalid Date comparison bypassed the past-event filter. Added Number.isNaN() validation to the filter and a navigation guard on the EventCard onPress.

EXACT ROOT CAUSE

The friendEvents filter at user/[id].tsx line ~568 did:
  const relevantTime = event.endTime ? new Date(event.endTime) : new Date(event.startTime);
  if (relevantTime < now) { return false; }

When event.startTime is undefined/empty/malformed, new Date(undefined) produces Invalid Date.
In JS: Invalid Date < new Date() === false — so invalid events PASS the filter.
These render with "Invalid Date" text. Tapping navigates to event detail which shows "Event details hidden" for ended/inaccessible events.

2. Filter: allFriendCalendarEvents (excludes circle-only) → friendEvents memo (excludes past + NOW invalid dates)
3. Event card render: friendEvents.map → EventCard component (line 156)
4. Card tap → navigation: EventCard onPress → router.push(`/event/${event.id}`)

FIX

Symptom A (Invalid Date cards):
- friendEvents filter now checks Number.isNaN(startMs) first — events with missing/malformed startTime are excluded
- endTime fallback also uses Number.isNaN() instead of truthy check
- Events with invalid dates never reach the render path

Symptom B (Dead-end navigation):
- EventCard onPress guards: if (!event.id || Number.isNaN(startMs)) return
- Belt-and-suspenders — the filter should catch all invalid events, but the nav guard prevents any edge case

BACKEND CONTRACT NOTE

Backend GET /api/friends/:id/events returns ALL events associated with the friendship, including ended events and events with null/undefined/malformed date fields. This is a backend issue — the endpoint should filter out ended events and validate date fields before returning. Client-side filter is the correct immediate fix; backend fix should be tracked separately in my-app-backend.
