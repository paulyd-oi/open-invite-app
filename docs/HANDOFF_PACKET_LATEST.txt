HANDOFF PACKET — Discover Avatar Stack Fix
Date: 2026-03-21

SUMMARY

Fixed Discover card attendee area showing only a +N count badge instead of real profile photos. Added host-user fallback so at least one real avatar always renders when attendeeCount > 0.

EXACT FILES CHANGED

1. src/app/discover.tsx
   - Attendee avatar stack builder (~line 799-813): added host fallback + dedup

ROOT CAUSE

The feed API endpoints (/api/events/feed, /api/events) return joinRequests as either undefined or an empty array — the backend does not populate this field for list/feed queries. The previous code built its avatar list exclusively from joinRequests filtered to status==="accepted". With zero accepted entries, the avatars array was empty, so only the +N overflow badge rendered (remaining = attendeeCount - 0 = attendeeCount). The host user (event.user), which IS always populated by the feed API with { id, name, image }, was never used as a fallback.

FIX APPLIED

Avatar list builder now:
1. Tries joinRequests (accepted, with user data) first — uses real attendee photos when available
2. Falls back to event.user (host) when joinRequests is empty — guarantees at least one real profile photo
3. Dedupes by user id to prevent the host appearing twice if also in joinRequests
4. Slices to max 3 displayed avatars
5. +N badge shows remaining = attendeeCount - displayed.length (only when > 0)

Result: every card with attendeeCount > 0 now shows the host's real profile photo as the first avatar circle, with +N for overflow. When joinRequests is populated (future backend change or different endpoint), real attendee photos render automatically.
