HANDOFF PACKET — Discover Truth Sync V1
Date: 2026-03-21

SUMMARY

Fixed two Discover tab data sync issues: (1) Events list empty/stale for second account due to refetchOnMount: false preventing stale-data refetch on navigation return, (2) Saved tab not reflecting cross-surface save/unsave because save mutation used narrow invalidation key and attending query wasn't in the focus refetch list.

EXACT ROOT CAUSE

Both symptoms trace to the same configuration cluster:

1. All three Discover queries (feedPopular, myEvents, attending) had `refetchOnMount: false`. TanStack Query's default behavior is `refetchOnMount: true` — when data exists but is stale, refetch on mount. By setting this to false, stale or invalidated data was never refetched when the user navigated back to Discover from another screen.

2. Save mutation from Discover invalidated only `eventKeys.feedPopular()` (specific key). Cross-surface saves from event detail invalidate `eventKeys.feed()` (parent) which prefix-matches feedPopular. But the Discover-local save was narrower than necessary and didn't invalidate `attending`.

3. The `attending` query was not included in the `useLiveRefreshContract` refetch list, so focus/foreground triggers didn't refresh it.

4. `staleTime: 30_000` (feed) / `60_000` (attending) meant data could be served from cache for up to 30-60 seconds even when outdated.

FIX

Frontend-only. Three changes:

A. Query config: Removed `refetchOnMount: false` from all three Discover queries. TQ default `true` means stale data refetches when user navigates back to Discover. This is the primary fix for both symptoms.

B. Save invalidation: Broadened to use parent `eventKeys.feed()` key (prefix-matches feedPopular + feedPaginated) plus `eventKeys.attending()`. Ensures all feed projections update after save.

C. Focus refetch: Added `refetchAttending` to useLiveRefreshContract so tab focus / foreground resume refreshes attending data alongside feed and myEvents.

BACKEND CONTRACT NOTE

The backend feed endpoint (/api/events/feed?visibility=open_invite) returns events appropriately scoped. For accounts with no social graph, the feed may return fewer or zero events — this is correct backend behavior (discovery is social-graph-scoped). The frontend issue was that even when the backend returned correct data, the stale cache configuration prevented it from being displayed. No backend changes needed for this fix.

One remaining backend observation: the feed endpoint does not populate joinRequests with attendee user data (the joinRequests array is empty/undefined in feed responses). This is why the Discover card avatar stack falls back to the host user photo. A backend enhancement to include top-N attendees in the feed response would improve the avatar stack, but is NOT required for this truth-sync fix.
