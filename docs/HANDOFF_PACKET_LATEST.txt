SUMMARY

Fixed friend request rows being non-functional on the Friends > People tab. Row taps silently did nothing because navigation used request.sender?.id (optional, often undefined from backend) instead of request.senderId (required, always present). Also removed refetchOnMount: false from the requests query to prevent stale data.

EXACT ROOT CAUSE

FriendsPeoplePane.tsx line 245: `const senderId = request.sender?.id`

The `sender` object is `.optional()` in the Zod schema (shared/contracts.ts line 388). When the backend doesn't populate the full sender relation, `request.sender` is undefined. The navigation guard at line 270 (`if (senderId)`) evaluates to false, silently blocking the tap.

Meanwhile, `request.senderId` (line 384 in schema) is a required string field — always present. It was never used for navigation.

Secondary: `refetchOnMount: false` on the friend requests query (friends.tsx line 866) meant stale/already-accepted requests persisted in the cache after navigating away and back.

FIX

Symptom A (row tap does nothing):
- Changed senderId extraction to use `request.senderId` (required field) as primary, `request.sender?.id` as fallback
- Navigation now always works when the request has a valid senderId (which it always does per schema)

Symptom B (stale/phantom request rows):
- Removed `refetchOnMount: false` and `refetchOnWindowFocus: false` from the friend requests query
- Stale data now refetches when navigating back to the friends tab
- placeholderData kept so UI doesn't flash empty during refetch

Accept/decline mutations:
- Were already correctly wired — they use `request.id` not sender info
- Optimistic removal, error rollback, post-mutation invalidation all intact
- No changes needed

BACKEND CONTRACT NOTE

Backend GET /api/friends/requests returns `sender` as optional in the response. When not populated:
- Row shows "Unknown" name and default avatar (existing fallbacks in FriendRequestCard)
- Navigation still works via `senderId` (always present)

Recommendation for my-app-backend: Always populate `sender: { id, name, image }` in the friend requests response. The `senderId` field ensures navigation works, but missing sender data degrades the UX (no name, no photo).
