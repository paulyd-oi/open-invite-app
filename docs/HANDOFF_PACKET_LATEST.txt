SUMMARY

Added filter chips to the notifications/activity feed so users can distinguish between notification types. Four filters: All (unfiltered), Events (invites, joins, comments), Friends (requests, accepted), Reminders. Chips appear above the FlatList as a horizontal ScrollView. "All" preserves current behavior with zero regression.

CURRENT STATE

The ActivityFeed component (used in both the full-screen activity.tsx route and embedded in Friends tab Activity pane) rendered all notifications in a flat list with no filtering. Each notification already has a `type` field with well-defined values and per-type icon/color config — the data supported filtering, only the UI was missing.

RUNTIME PATH
1. Notification data load: ActivityFeed → usePaginatedNotifications → GET /api/notifications/paginated?limit=30
2. Notification list render: FlatList with filteredNotifications (memo'd from uniqueNotifications + activeFilter)
3. Filter UI: horizontal ScrollView of Pressable chips as ListHeaderComponent

FIX

Filter chip design:
- Horizontal ScrollView with flexGrow: 0 to constrain height
- Each chip: rounded-full pill with icon + label
- Active chip: themeColor background, white text
- Inactive chip: colors.surface background, colors.border border, colors.textSecondary text
- Haptics.selectionAsync on tap

Filter logic:
- FILTER_TYPE_MAP maps filter key → Set of notification type strings (or null for "all")
- filteredNotifications memo: if typeSet is null (all), return all; otherwise filter by Set.has()
- Chips only visible when uniqueNotifications.length > 0 or filter is not "all"

Filter categories:
- All: null (no filter)
- Events: event_invite, event_join, event_comment
- Friends: friend_request, friend_accepted
- Reminders: event_reminder, reminder

Unmatched types (achievement, referral, etc.) appear in "All" only. This is intentional — they're rare and don't warrant their own filter.

BACKEND CONTRACT NOTE

Backend GET /api/notifications/paginated returns `type: string` on every notification. Types observed: friend_request, friend_accepted, event_invite, event_reminder, event_join, event_comment, achievement, referral. The type field is reliable and well-populated. No backend changes needed.
